### Step 7 failure — root cause

```
error_message: Worker step "setup_mail_domains" failed:
  Failed to fix @ A/MX for S2 domain slaunter.info on S1:
  SSHCommandError: Command failed with exit code 3:
  export PATH=/usr/local/hestia/bin:$PATH && v-list-dns-records admin slaunter.info plain
```

Thrown at [pair-provisioning-saga.ts:1085](src/lib/provisioning/pair-provisioning-saga.ts:1085) from inside PATCH 10c at [pair-provisioning-saga.ts:1002-1087](src/lib/provisioning/pair-provisioning-saga.ts:1002). The outer loop iterates `server2Domains` and for each, the inner loop runs the "fix @ A/MX" block on BOTH halves:

```ts
for (const domain of server2Domains) {
  for (const [sshConn, label] of [[ssh1, 'S1'], [ssh2, 'S2']] as [typeof ssh1, string][]) {
    // v-list-dns-records admin ${domain} plain
    // ↑ exit 3 on S1 because slaunter.info is S2-primary (BIND slave only on S1 post-PR #4)
  }
}
```

**Root cause: pre-existing saga regression from PR #4 (HL #101), orthogonal to PR #12 / HL #111.**

Before PR #4, both halves were BIND primaries for every zone (dual-primary) and Hestia knew about every zone on both halves — `v-list-dns-records admin <any-zone>` worked from either side. PR #4 (`7839c9c`, merged 2026-04-19) partitioned zones via `computeZonePartition` and installed slave zones on the peer via `installSlaveZones`. Post-PR #4, S2-primary zones exist on S1 only as BIND slaves in `/etc/bind/named.conf.cluster` — they are NOT Hestia-managed on S1, and `v-list-dns-records admin <s2-primary-zone>` returns exit 3 "dns domain X doesn't exist".

PATCH 10c dates to the dual-primary era. P13 launta.info shipped via backfill scripts (`scripts/launta-axfr-backfill.sh`) that converted an already-running dual-primary pair to AXFR — P13 never exercised the fresh-saga PATCH 10c path post-PR #4. P14 is the first fresh saga run through this code on the new AXFR architecture and surfaces the regression immediately on the first S2 sending domain (slaunter.info — alphabetically first in `server2Domains`).

**Suggested fix (NOT implemented per prompt's "no hotfix" rule):** restrict PATCH 10c's S2-domain fix to the owning server only (`ssh2` for `server2Domains`). AXFR/NOTIFY from PR #4/#8 propagates the `@ A` and `@ MX` rewrites from S2 → S1 automatically (verified on launta.info in PR #8 merge evidence). The `for [[ssh1, 'S1'], [ssh2, 'S2']]` mirror-writes pattern predates AXFR and is now redundant-and-broken. The sibling block at [pair-provisioning-saga.ts:1095+](src/lib/provisioning/pair-provisioning-saga.ts:1095) (PATCH 13f1, "Fix mail/webmail A records for S2 domains on BOTH servers") likely has the same regression and needs the same audit before any re-run.
