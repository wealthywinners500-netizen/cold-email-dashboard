### C. Env-key naming asymmetry — credentials are DB-driven, not env-driven

Agent finding (key insight): no `process.env.IONOS_*`, `process.env.LINODE_*`, or `process.env.NAMECHEAP_*` reads in `src/`. Credentials come from encrypted DB tables (`vps_providers` and `dns_registrars`), decrypted via the same AES-256-GCM `ENCRYPTION_KEY` flow as `ssh_credentials.password_encrypted`.

| Asymmetry | Worker-side direct env read | Severity per Dean's lock |
|---|---|---|
| F-19 IONOS (`API_KEY` vs `API_PREFIX`+`API_SECRET`) | NO — DB-driven | **P2** holds |
| F-20 LINODE (`API_TOKEN` vs `API_KEY`) | NO — DB-driven | **P2** holds |
| F-21 NAMECHEAP (Vercel-only, worker has none) | NO — DB-driven | **P2 holds** (per Dean's escalation rule: only escalate to P1 if worker-side code reads them; worker doesn't read directly) |

The Vercel/worker env-key NAMES are largely informational since the actual credentials live in DB. F-19/F-20/F-21 effectively demote to **P2 cleanup** — orphan env vars on Vercel/worker that nothing reads. Phase 8 candidate: prune unused env-vars (low priority, no runtime impact).
