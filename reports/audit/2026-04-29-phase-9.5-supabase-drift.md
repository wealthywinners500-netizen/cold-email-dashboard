### F-NEW-9.5-SUPABASE-2 — Migration tracker drift (P3)

**Trigger:** V4 plan §9.5 referenced migrations `008_system_health` / `009_b14_hardening` / `012_dry_run_enums` as "the keepers." Querying `supabase_migrations.schema_migrations` returns only 2 entries:
```
021|021_dbl_resweep
008|atomic_counters
```

**Discrepancy:**
- V4 said "008 = system_health" — actual is `008_atomic_counters`. Different migration with the same number.
- 009, 012, and 21 other migrations are NOT in the tracker, despite the schema having tables clearly added by them (`worker_heartbeats`, `dbl_sweep_runs`, etc.).
- Local migration files in `dashboard-app/supabase/migrations/` likely have the canonical sequence (`001_*` through `021_*`), but only 2 of them recorded a row in the `supabase_migrations.schema_migrations` table.

**Likely cause:** migrations were applied via direct `psql` or Supabase Studio SQL editor (not via `supabase db push` which writes to the tracker). This is a documented historical pattern — see `dashboard-app/reports/2026-04-22-hl-1-80-audit-decisions.md` and the recent migration tracker repair work (`reports/2026-04-27-migration-tracking-repair.md`).

**Resolution:** Out of audit scope to fix. Migration-tracker repair is its own workstream item — `dashboard-app/reports/2026-04-27-migration-tracking-repair.md` already documents the gap and has a remediation path. Reference in FIX-BACKLOG cross-link.
